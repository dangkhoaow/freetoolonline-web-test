The current time in milliseconds is a single integer the browser already knows: Date.now() returns a 13-digit number that counts milliseconds since the Unix epoch at 1970-01-01T00:00:00 UTC. The live tool prints that value without any network round trip, which is why the read is instant and why nothing about the request leaves the device. Most readers arrive here while debugging a log line, comparing two timestamps from different systems, or sanity-checking that a JSON field is in the right shape - the value on this page is the same shape a server returns and a log line records, just rendered as plain text.

The most common mistake is the 10-digit-versus-13-digit mix-up. A 10-digit number is seconds since the epoch; a 13-digit number is milliseconds. Mixing the two produces dates that look decades off - a 13-digit value parsed as seconds points somewhere in the year 58000, a 10-digit value parsed as milliseconds points to 1970 plus a few hours. The fix is mechanical: count the digits. If you have a value and are not sure which unit it is in, the count-of-digits rule resolves it without a calculator.

Once you have the millisecond value, the next decision is rendering. A timestamp is not a date until something formats it for a human reader; the milliseconds to date converter renders the same integer in UTC and in your local timezone side by side, so you can compare the two and pick the column that matches the destination. For the deeper timezone question (why the same number reads 15:00 in UTC, 22:00 in Bangkok, and 08:00 in Los Angeles), read milliseconds to date: UTC vs local time.
