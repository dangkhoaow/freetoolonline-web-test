ZIP password types - strong vs weak, explained (when an online remover can help)

Last reviewed 2026-05-04. A 30-second decision guide before you upload a password-protected ZIP to Remove ZIP Password: which kinds of ZIP encryption can an online tool actually remove, and which kinds are computationally infeasible to crack? The honest answer turns on two things: (1) the encryption algorithm baked into the ZIP (legacy ZIP 2.0 traditional encryption versus WinZip AES-256), and (2) the strength of the password the file's creator chose. The pairing decides whether the file is recoverable in seconds, in hours, or essentially never. This guide names three concrete cases, with what to do for each, so you do not upload a 200 MB archive to a tool that cannot help you.

30-second decision. Open the ZIP in any archive manager (7-Zip, WinRAR, the macOS Archive Utility, or the Files app on Windows). Read the encryption label on the locked entry. "ZipCrypto" / "Traditional encryption" / "ZIP 2.0" with a short numeric or dictionary-word password → Remove ZIP Password typically lands the answer in seconds-to-minutes. "AES-256" (or "AE-2", the WinZip AES variant) with a strong 12+ character mixed-character password → an online remover cannot help; the right next step is to ask the file's creator for the password, or use the original tool that created the ZIP. "AES-256" with a short or dictionary password → possibly recoverable depending on the password, but slower than ZIP 2.0; try Remove ZIP Password and see.

Case 1 - Legacy ZIP 2.0 (traditional encryption): recoverable in seconds-to-minutes for short / numeric / dictionary passwords

"ZIP 2.0 traditional encryption" (also called "ZipCrypto" by 7-Zip and "ZIP encryption" by WinZip) is the original ZIP password format introduced in PKWARE's PKZIP 2.0 in 1989. It is the default password format for the built-in ZIP support on older Windows ("Send to compressed (zipped) folder" on Windows 7 / 8 / 10) and for many web-app file-export ZIPs. The encryption algorithm is a stream cipher derived from the password; the format has well-known cryptographic weaknesses that allow an attacker with even a small piece of the original plaintext to recover the keystream without brute-forcing the password. In practice, however, "remove ZIP password" tools rely on plain dictionary + brute-force search against the password itself - which works fast because (a) the ZIP 2.0 key derivation is cheap (one CRC-32 + a few bit operations per password attempt, not a deliberately slow KDF like PBKDF2) and (b) most passwords on this format are short, numeric, or dictionary words.

  • When the online remover wins fast: the password is 1-8 characters, all-numeric (a date, a phone number, a postcode), an English / common-language dictionary word, or a documented default like "123456" / "password" / the recipient's name. Remove ZIP Password typically lands the answer in seconds-to-minutes for these.
  • When the online remover slows down: the password is 9-12 characters mixed-case alphanumeric. The brute-force search space grows by ~70x for every additional character, so the same machine that finished a 6-char password in 2 seconds takes weeks for a 10-char password. The remover may time out before finding the answer.
  • How to tell you have ZIP 2.0: in 7-Zip, right-click the locked entry → Properties → "Method" column shows ZipCrypto (or "ZIP encryption"). In WinRAR, the file properties show "Encrypted: Yes" and the encryption details say "ZIP 2.0 (legacy)". On macOS Archive Utility, the locked icon plus the file's PKZIP version (PKZIP 2.0 in the central directory) indicates this format; you can verify by running unzip -v archive.zip in Terminal and looking for the encryption mode in the central-directory header.
  • What to do with the recovered file: once Remove ZIP Password returns the unlocked archive, the ZIP file inside no longer has the legacy encryption, so re-zipping it later in a modern tool with AES-256 is the right next step if you need to keep the file confidential.

Case 2 - WinZip AES-256 with a strong password: not recoverable by any online remover

WinZip AES-256 (introduced as "AE-2" in the WinZip 9.0 ZIP specification in 2003 and adopted by 7-Zip, WinRAR, modern macOS, and most modern Linux ZIP tools) uses the Advanced Encryption Standard at a 256-bit key size. The key is derived from the password through PBKDF2 with HMAC-SHA1 and 1,000 iterations - a deliberately slow key-derivation function. The combination defeats brute-force search: a strong password (12+ characters, mixed letter case, digits, and symbols, not a dictionary word) creates a search space of roughly 10^24 candidates that no single machine, GPU rig, or online tool can iterate through in any reasonable time. The cryptography is the same algorithm used to protect classified government data; it is computationally secure against brute force at the password sizes most users actually pick.

  • When this case applies: the file's creator deliberately chose AES-256 in 7-Zip ("Encryption method → AES-256"), WinZip ("Use AES encryption"), WinRAR (the AES default for ZIP since 5.x), or macOS's built-in archiving (which uses AES-256 by default since macOS 10.13). The password is 12+ characters and not a dictionary word.
  • What an online remover cannot do: an online "remove ZIP password" tool runs a brute-force search against the password. With AES-256 + a strong password the search would take longer than the age of the universe on any hardware that exists today. The tool will run, time out, and return "not recoverable". The cryptography did its job - this is the expected outcome, not a tool failure.
  • How to tell you have AES-256: in 7-Zip, the "Method" column shows AES-256. In WinRAR, the encryption details say "AES-256" or "AE-2". unzip -v archive.zip on Linux / macOS shows the encryption mode in the central-directory header; the WinZip AE-2 spec uses a specific compression-method code (99) in the local file header that unzip reports.
  • What the right next step is: ask the file's creator for the password, use the password-recovery feature of the original archive tool that created the file (some tools cache previously used passwords), check the email / chat / document where the password was originally communicated, or - if the password is genuinely lost - treat the file as unrecoverable and request a fresh export from the source. The companion guide Recover a corrupt ZIP file - options covers the ZIP-recovery flow when the issue is corruption rather than a forgotten password; the two flows are different.

Case 3 - WinZip AES-256 with a weak / short / dictionary password: possibly recoverable, slower than ZIP 2.0

The middle case is the tricky one. The ZIP file uses AES-256 (modern, strong cryptography), but the password the creator picked is short (6-8 characters), all-numeric, or a dictionary word. The PBKDF2-with-1,000-iterations KDF makes each password attempt about 1,000 times slower than a ZIP 2.0 attempt - so the same brute-force search that finished in 2 seconds on a ZIP 2.0 password takes ~30 minutes on AES-256. Dictionary attacks against a common English / common-language word still finish in minutes-to-hours; brute-force against a short numeric password (1-8 digits = 10^8 = 100 million candidates) finishes in hours-to-a-few-days on consumer hardware. Remove ZIP Password attempts this search and may or may not return the answer within its time budget.

  • When the online remover may still help: the password is suspected to be 1-8 digits (a date, a postcode, a four-digit PIN extended to 8) or a single common dictionary word. The remover tries the high-probability candidates first.
  • When the online remover times out: the password is 9+ characters, mixed-case, includes symbols, or is a passphrase (multiple dictionary words concatenated). At that point the file is in Case 2 and an online remover is not the right tool.
  • What to expect: Remove ZIP Password reports either the recovered password (Case 1 / weak Case 3 outcome) or a "not recoverable within time budget" status (strong Case 3 / Case 2 outcome). The tool does not pretend to crack passwords it cannot.

Decision flow: from a locked ZIP to the right next step

Step 1 - identify the encryption type. Open the ZIP in 7-Zip, WinRAR, or run unzip -v archive.zip in Terminal. Read the encryption label. ZipCrypto / "ZIP 2.0" / "Traditional" → Case 1. AES-256 / "AE-2" → Cases 2 or 3, decided by password strength.

Step 2 - estimate the password strength. If you have any guess at all about the password (a date you used, a project codename you might have picked, a recipient's name) you fall into Case 1 / weak Case 3 territory and an online remover is worth trying. If the file came from a security-conscious sender who chose a 16-character random passphrase, you fall into Case 2 territory and an online remover will not help.

Step 3 - act on the case. Case 1 or weak Case 3 → upload to Remove ZIP Password; expect an answer in seconds-to-hours depending on length. Case 2 (or strong Case 3) → do NOT upload; the right next step is to recover the password from where it was originally stored (email, password manager, documentation), ask the sender, or - if the file is genuinely orphaned from its password - treat it as unrecoverable and request a fresh export. If the issue is a corrupt ZIP rather than a forgotten password, see Recover a corrupt ZIP file - options.

Step 4 - re-protect after recovery. If Remove ZIP Password returned an unlocked archive and you still need confidentiality, re-zip the contents in Compress, ZIP File and Folder or in 7-Zip with AES-256 and a strong passphrase you record in a password manager. The recovered file is in legacy ZIP 2.0 by design (that is what the remover targeted); leaving it in that format keeps it weakly protected.

Why the math works the way it does (the short version)

The fundamental difference between ZIP 2.0 and AES-256 is the cost-per-attempt of the password search. ZIP 2.0's key derivation is a few CRC-32 operations and bit shifts - the encoder built the format to be fast on a 1989 PC, before deliberately-slow key-derivation functions were a thing. A modern CPU rips through millions of ZIP 2.0 password attempts per second. AES-256 in WinZip uses PBKDF2-HMAC-SHA1 with 1,000 iterations - 1,000 hash computations per attempt - so the same modern CPU manages thousands of AES-256 attempts per second, not millions. That 1,000x slowdown is the whole point: it makes brute-force search ~1,000x more expensive without affecting normal users (who only enter the password once and do not notice the millisecond-scale derivation cost). Pair the slow KDF with a strong password (large search space) and the file is computationally secure. Pair fast KDF with a short password (small search space) and the file is fast to recover. The companion guide PDF password types - owner vs user covers the analogous structure on the PDF side; the underlying lesson is the same in both formats - encryption strength + password strength multiply, and a weakness in either makes the file recoverable.

For everything else - one ZIP, one decision, the right next step - the four-step flow above lands the answer in 30 seconds. Remove ZIP Password is the right tool when the math says the file is recoverable; it honestly returns "not recoverable" when the math says it is not, instead of pretending to crack what it cannot.
