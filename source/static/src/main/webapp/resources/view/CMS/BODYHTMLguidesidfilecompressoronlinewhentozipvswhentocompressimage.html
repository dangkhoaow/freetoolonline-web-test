Kompresor file online - kapan ZIP sebuah folder, kapan mengompres satu gambar?

Jawaban 30 detik. Apakah Anda mencoba membuat folder file lebih kecil untuk dikirim atau disimpan? Buka Zip File - ia membundel setiap file dalam folder ke satu arsip ZIP, opsional dengan password. Apakah Anda mencoba membuat satu gambar lebih kecil agar berhenti menggembungkan email atau halaman yang lambat? Buka Compress Image - ia mengencode ulang JPEG / PNG dengan ukuran byte lebih kecil sambil mempertahankan kualitas terlihat. Mengapa "kompresor file" adalah dua operasi berbeda (dan mengapa keduanya mengunggah ke server)

Frase "kompresor file" mencakup dua tugas pembaca berbeda yang memiliki satu kesamaan - output lebih kecil dari input - tapi selain itu hampir tidak berbagi apa pun. Tugas kompresi folder mengambil banyak file (atau satu file plus struktur foldernya) dan membundelnya ke satu kontainer arsip yang lebih cepat dikirim dan lebih mudah disimpan. Zip File adalah alat untuk tugas itu: ia mengompres folder dan file ke arsip ZIP online dan menambahkan perlindungan password opsional sebelum download, jadi satu upload mengembalikan satu .zip. Tugas gambar tunggal mengambil satu JPEG atau PNG dan mengencode ulang pixel pada target kualitas yang lebih rendah sehingga file menyusut - dimensinya tetap sama tapi file lebih kecil di disk. Compress Image adalah alat untuk tugas itu: tarik gambar, terima target kualitas default, dan unduh versi yang lebih kecil. Dua input berbeda, dua output berbeda, dua alat berbeda.

Kedua rute mengunggah input Anda ke layanan AWS freetoolonline - langkah konversi berjalan di sisi server dan hasilnya diunduh kembali ke browser Anda. File output disimpan di S3 dan dirotasi setelah jendela retensi standar layanan. Jika Anda secara khusus membutuhkan jalur yang tidak mengunggah, panduan ini bukan aturan yang tepat untuk kendala itu - lihat alat online gratis yang berfungsi tanpa mengunggah file untuk rute tanpa-upload (yang merupakan kelompok berbeda dan tidak termasuk /zip-file.html atau /compress-image.html).

Aturan perutean 1 - input Anda adalah folder file: buka Zip File

Jika input Anda lebih dari satu file - atau satu file plus folder tempatnya - Zip File adalah alatnya. Sinyal konkret bahwa Anda berada dalam kasus ini:

  • Batas lampiran email memblokir Anda mengirim folder berisi kuitansi, foto, atau draft kontrak. ZIP meruntuhkan folder menjadi satu .zip yang dianggap klien email sebagai satu lampiran tunggal.
  • Upload "tarik folder di sini" di layanan lain menolak folder karena hanya menerima file - kontainer ZIP membuat folder tampak seperti file.
  • Anda ingin membagikan proyek (situs kecil, folder pekerjaan rumah, contoh kode) sebagai satu link bukan banyak. Penerima membuka zip dengan OS atau dengan Unzip File di browser.
  • Anda ingin menambahkan password sebelum mengirim. Zip File mengekspos perlindungan password opsional pada alur yang sama; penerima dengan password menggunakan Remove ZIP Password atau OS untuk membukanya. Kit ZIP lengkap (kompres / unzip / pasang password / hapus password) tergulung di ZIP Tools.

Alasan: ZIP dirancang persis untuk tugas pembaca "banyak file, satu amplop". Kompresi itu sendiri adalah efek samping untuk banyak campuran input (foto dan video yang sudah dikompres tidak menyusut banyak di kontainer ZIP), tetapi pembundelan - satu amplop, satu upload, satu download - adalah nilai yang diberikan halaman terindeks. Panduan pendamping Kompresor file vs ZIP - mana yang dipilih menempuh perbandingan berdampingan; Cara mengompres folder menempuh alur klik demi klik. Jika input Anda adalah satu gambar (bukan folder), ini bukan aturan Anda - lompat ke aturan 2.

Aturan perutean 2 - input Anda satu gambar dan Anda ingin lebih sedikit kilobyte: buka Compress Image

Jika input Anda satu JPEG atau PNG dan output yang Anda inginkan adalah gambar yang sama dengan ukuran byte lebih kecil, Compress Image adalah alatnya - bukan ZIP. Sinyal konkret:

  • Blog atau halaman produk dimuat lambat di mobile karena foto hero dua megabyte. Compress Image mengencode ulang JPEG pada target kualitas lebih rendah (tipikal: kualitas 75-85 pada slider JPEG 4-sumbu yang setara) dan membawanya di bawah beberapa ratus kilobyte tanpa mengubah gambar secara terlihat.
  • Batas lampiran email menolak satu foto besar (mis. HEIC iPhone sudah dikonversi ke JPG 6 MB). Compress Image menurunkan ukuran byte; jika format HEIC, jalankan melalui HEIC to JPG dulu. Trade-off (kualitas terlihat vs ukuran file) ada di Kompres JPEG tanpa kehilangan kualitas - kualitas vs ukuran.
  • Alat web atau platform sosial menolak foto karena melebihi batas byte per gambar. Dimensi gambar baik; hanya ukuran byte adalah pemblokirnya. Compress Image menargetkan persis itu.

Alasan: meletakkan satu JPEG dalam ZIP hampir tidak pernah menyusutkannya, karena byte JPEG sudah dikompres pada tingkat codec - kontainer ZIP hanya menambahkan beberapa byte amplop. Rute gambar tunggal mengubah target kualitas codec itu sendiri, yang merupakan satu-satunya operasi yang menghasilkan file lebih kecil yang terukur. Jika gambar terlihat buram setelah alat khusus, alur diagnostik ada di JPG terkompres terlihat buram - tiga penyebab. Jika Anda memiliki banyak gambar sekaligus, lihat aturan 3.

Aturan perutean 3 - input campuran (screenshot, PDF, banyak foto): pilih berdasarkan bentuk output

Jawaban dua-alat di atas mencakup kasus bersih. Kasus campuran membutuhkan satu pertanyaan ekstra - dalam bentuk apa Anda ingin output?

  • Banyak foto, Anda ingin satu amplop untuk dikirim: ZIP dulu via Zip File. Setiap foto di dalam ZIP mempertahankan ukuran byte-nya sendiri; ZIP hanya menghemat Anda satu unggahan slot lampiran. Jika foto-foto secara individu terlalu besar, kompres masing-masing dengan Compress Image dulu, lalu ZIP versi terkompresinya.
  • Banyak foto, Anda ingin masing-masing lebih kecil secara individu: jangan ZIP - kompres masing-masing dengan Compress Image. ZIP tidak mengencode ulang JPEG. Jika Anda perlu memproses banyak gambar secara batch dengan satu upload, yang paling cocok adalah Photo Editor untuk alur batch interaktif, atau ImageMagick Online dengan pipeline yang diketik (dibahas di ImageMagick Online vs Task-Specific Tools - mana yang dipilih).
  • Satu PDF yang terlalu besar untuk email: mengompres PDF adalah operasi berbeda dari mengompres gambar. Rute khusus terdekat di situs ini adalah Compose PDF untuk rakit ulang; aturan optimisasi khusus PDF (subsetting font, downsampling gambar dalam PDF) tidak diekspos oleh Zip File atau Compress Image, jadi tidak ada alat yang berlaku secara bersih.
  • Screenshot (PNG): mengompres screenshot pada tingkat codec biasanya berarti mengonversi PNG ke JPG dulu karena codec PNG mempertahankan setiap pixel persis. Compress Image menangani itu; latar belakang pilihan format ada di JPG vs PNG untuk web.

Alasan: tipe input dan bentuk output menentukan alat mana. Kesalahan yang harus dihindari adalah "saya akan ZIP semuanya" ketika yang sebenarnya Anda inginkan adalah foto-foto lebih kecil secara individu, atau "saya akan kompres gambar saja" ketika yang sebenarnya Anda inginkan adalah satu amplop berisi beberapa file. Pilih berdasarkan bentuk output, bukan kata di kotak pencarian.

Aturan perutean 4 - file sangat besar atau sensitif: pikirkan apakah harus mengunggah sama sekali

Baik Zip File dan Compress Image mengunggah input Anda ke layanan AWS freetoolonline. Itu jalur yang benar untuk hampir setiap tugas pembaca alat browser, tetapi dua kasus memanggil jawaban perutean berbeda. Pertama, input sangat besar (folder foto RAW dalam gigabyte, file video sepanjang satu jam dalam folder, arsip proyek yang sudah melebihi beberapa ratus megabyte) lambat untuk diunggah dan diunduh bahkan di koneksi yang baik - jika Anda mengendalikan mesin desktop, alat ZIP bawaan OS Anda (klik kanan - "Compress" di macOS, klik kanan - "Send to - Compressed folder" di Windows, zip di Linux) lebih cepat dari ujung ke ujung karena byte tidak pernah meninggalkan disk Anda. Kedua, file yang tidak dapat Anda unggah karena alasan kerahasiaan (imaging medis, bukti hukum, karya kontraktual, kode sumber di bawah NDA) perlu tetap lokal - alat ZIP tingkat OS adalah jalur yang benar bahkan jika file kecil.

Trade-off memiliki bentuk yang sama dengan keputusan perutean video dan gambar yang setara: FFmpeg online vs FFmpeg lokal - kapan masing-masing menang dan ImageMagick Online vs Task-Specific Tools - mana yang dipilih keduanya menempuh batas "online vs lokal" secara rinci. Batasnya sama di sini: rute online menang dalam kenyamanan untuk input ukuran normal, tidak sensitif; rute lokal menang dalam kecepatan dan privasi untuk yang sangat besar atau sensitif.

Aturan perutean 5 - output salah atau upload macet: troubleshoot sebelum menjalankan ulang

Jika Zip File atau Compress Image menerima upload tetapi tidak ada download muncul dalam beberapa menit, jangan upload ulang. Mengunggah ulang file yang sama tidak memperbaiki penyebab dasar; ia mengantrekan pekerjaan kedua yang menabrak tembok yang sama. Macet umum memiliki bentuk yang sama di kedua alat: batas ukuran input tercapai saat upload, tab browser disegarkan selama pipeline konversi, atau kendala format output tidak terpenuhi (mis. mencoba ZIP folder yang ukuran totalnya melebihi tutup upload).

Jika output adalah bentuk yang salah, Anda mungkin memilih alat yang salah. Jika .zip Anda tiba tetapi setiap foto di dalamnya masih besar, Anda menginginkan Compress Image dulu, lalu Zip File - .zip hanya membundel, ia tidak mengencode ulang. Jika JPG terkompres Anda tiba tetapi sekarang buram alih-alih hanya lebih kecil, target kualitas diatur terlalu rendah - alur diagnostik ada di JPG terkompres terlihat buram - tiga penyebab. Jika penerima tidak dapat membuka .zip, mereka mungkin perlu Unzip File di browser, dan jika ada password yang menghalangi Remove ZIP Password mencakup alur itu. Untuk segala lainnya - satu operasi umum, pengaturan default, alat yang tepat untuk bentuk input - keputusan perutean adalah salah satu dari empat aturan di atas: ZIP untuk folder, Compress Image untuk gambar tunggal, input campuran berdasarkan bentuk output, instalasi lokal untuk file sangat besar atau sensitif. Jawaban yang benar memakan 30 detik; jawaban yang salah memakan tambahan satu menit waktu upload yang tidak perlu Anda habiskan.
