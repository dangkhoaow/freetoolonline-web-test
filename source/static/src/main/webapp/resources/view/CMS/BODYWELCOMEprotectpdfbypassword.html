User password vs owner password: what each one actually restricts

PDF encryption ships two distinct password types and the one you pick changes what a reader can do with the file. A user password (open password) blocks opening the file at all; the reader must enter it before seeing any content. An owner password (permissions password) lets anyone open the file, but restricts actions: printing, copying text, annotating, filling forms, or extracting pages. For a confidential contract, set a user password so the file cannot be read without it. For a whitepaper where you want free distribution but need to block editing, set an owner password. You can also set both on the same file - one to open, one to edit. This tool supports either one or both; pick the combination that matches your sharing policy.


Choosing a PDF password that resists the common attacks

A password that takes a modern GPU more than a few hours to try every combination of is strong enough for most business documents. Target length is twelve characters or more, mixing cases, digits, and at least one punctuation mark. Avoid words from any dictionary, any date associated with the document (year published, contract date, employee hire date), and any string derived from the filename. Passphrases of four unrelated words remain the strongest memorable option for a PDF you need to share and then forget about. If the document is archival, consider encrypting the file inside a ZIP with a second, separate password - the PDF password blocks the viewer, the ZIP password blocks the byte-level file until the passphrase is entered.


Privacy and how this tool handles the password you set

Encryption runs in your browser. The PDF is loaded locally, the password you type is fed into the encryption routine on the page, and the encrypted copy is generated in memory. Nothing is uploaded to any server, and the password is never logged or transmitted. When you close the tab, both the file and the password are wiped from memory. That matters for NDA contracts, payroll exports, and medical records where posting the password to a cloud service would defeat the encryption's purpose from the start.
