Kompresor ZIP

Kompresor zip mengambil folder file dan menulis satu arsip .zip yang berisi mereka. Arsip lebih kecil dari jumlah file hanya saat file tersebut dapat dikompresi (teks biasa, kode sumber, gambar tidak terkompresi); format yang sudah terkompresi (JPG, MP4, MP3) biasanya tumbuh beberapa persen.

Panduan ini dalam fase draf awal. Panduan lengkap tugas pembaca - kapan memilih kompresi zip, apa yang dipertahankan dan tidak dipertahankan arsip, dan cara membuka hasilnya tanpa kehilangan nama file - tiba di pass penerbitan berikutnya. Sementara itu, alat zip di browser ada di /zip-tools/zip-file.html.

Kapan kompresi zip benar-benar menghemat ruang

Kompresi zip menghemat ruang hanya saat file input belum dikompresi oleh format mereka sendiri. Teks biasa, kode sumber, ekspor CSV, file log, bitmap tidak terkompresi, dan audio tidak terkompresi (seperti WAV) semuanya mengompresi dengan baik - folder berisi ini dapat menyusut setengah atau lebih di dalam .zip. Format yang sudah terkompresi - JPG, PNG, MP4, MP3, dan .zip apa pun yang ditempatkan di dalam .zip lain - biasanya tumbuh beberapa persen karena header arsip dan metadata per entri ditambahkan tanpa kompresi lebih lanjut yang mungkin pada muatan. Jika folder Anda didominasi foto atau video, harapkan .zip yang dihasilkan memiliki ukuran kira-kira sama dengan aslinya; manfaatnya di sana adalah menggabungkan banyak file menjadi satu arsip yang bisa diunduh, bukan penyusutan.

Di mana kompresor zip berbasis browser berada

Kompresor zip di browser di situs ini berada di /zip-tools/zip-file.html. Halaman panduan ini adalah tempat konteks keputusan berada - kapan kompresi zip benar-benar membantu, apa trade-off-nya, dan tipe file mana yang layak digabung. Halaman alat adalah tempat arsip itu sendiri dibuat: buka alat, tambahkan folder atau file yang dipilih, beri nama arsip, dan unduh file .zip yang dihasilkan. Kedua halaman melayani tugas pembaca yang berbeda (keputusan vs eksekusi) dan tidak menduplikasi satu sama lain.

Tipe file mana yang benar-benar menyusut di dalam .zip

Tiga aturan praktis menentukan apakah folder layak di-zip untuk transfer. Format teks biasa (kode sumber, ekspor CSV, file log) mengompresi paling kuat dan secara rutin kehilangan setengah atau lebih dari jumlah byte di dalam arsip. Bitmap tidak terkompresi (BMP) dan audio tidak terkompresi (WAV) berada di tengah dan masih menyusut tiga puluh hingga enam puluh persen. Apa pun yang sudah dikompresi format - JPG, PNG, MP4, MP3, atau .zip yang ditempatkan di dalam .zip lain - tumbuh satu hingga tiga persen di dalam arsip baru karena metadata per entri ditambahkan di atas byte muatan yang tidak dapat di-deflate lebih lanjut. Pertanyaan umum

Apakah .zip mempertahankan nama file dan struktur folder?

Ya. Arsip .zip mempertahankan nama file dan struktur direktori secara verbatim; pada perjalanan bolak-balik lintas sistem operasi, ia tidak mempertahankan kepemilikan file atau atribut yang diperluas.

Di mana saya membuat arsip di browser?

Kompresor zip berbasis browser di situs ini ada di /zip-tools/zip-file.html. Panduan ini adalah halaman konteks keputusan; halaman alat adalah tempat arsip dibuat.

Panduan ZIP terkait di situs ini

Jika Anda ingin pandangan yang lebih luas tentang bekerja dengan arsip .zip di browser, dua panduan saudara mencakup tugas pembaca terkait:

  • Cara mengekstrak file secara online (ZIP, RAR, 7z) - tugas kebalikan dari kompresi: membuka arsip yang seseorang kirim, memilih file mana yang akan diekstrak, dan memilih di mana di disk file yang diekstrak akan ditempatkan.
  • Cara mengurangi ukuran file .zip secara online - setelah Anda memiliki arsip, panduan ini mencakup pilihan yang benar-benar menyusutkannya lebih lanjut: menghapus file redundan, memilih apakah akan mengompresi ulang input yang sudah terkompresi, dan kapan membagi menjadi beberapa arsip lebih cepat daripada kompresi yang lebih ketat.
