Two practical limits shape what "MD5 decrypt online" can deliver. First, the MD5 converter at /md5-converter.html caps input at 990 characters when hashing and only accepts exactly 32 hex characters when looking up; inputs outside those bounds are rejected with an inline error before any request fires, because the constraints come straight from the MD5 spec (32-hex output) and the textarea size cap set on the page. Second, the To Text lookup runs against a dictionary - if the hash you paste was paired with a plaintext earlier on this browser (cached in localStorage) or in the server's dictionary, the original string comes back; if not, the lookup returns nothing, which is the expected behaviour rather than a tool failure.

The hashing path itself is a hybrid: server-side over HTTPS by default, with a local md5() fallback (loaded from CDN as md5.min.js) that fires automatically if the server call fails. Practically, that means generating a fingerprint from a known string succeeds even on a flaky connection - the To MD5 path is genuinely "decrypt-proof" in the cryptographic sense but always works for the legitimate one-way direction. If you arrived hoping to recover a forgotten password from an MD5 stored years ago, the right next click is pick an MD5 alternative, not another lookup attempt. The fingerprint that MD5 produced years ago is still a fingerprint, not an encrypted copy of the password.
