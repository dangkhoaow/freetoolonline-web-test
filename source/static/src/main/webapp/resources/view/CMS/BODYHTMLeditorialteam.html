Editorial Team

Last reviewed 2026-04-24.

Free Tool Online is built and maintained by a small editorial team that writes the tool copy, vets every tool's behavior in production browsers, and reviews the privacy and security posture of anything that touches a user's file. This page documents who reviews what, how contributions are vetted, and how the editorial cadence works - the same E-E-A-T signals that Google's Helpful Content System and Search Quality Rater Guidelines ask publishers to surface.

Editorial principles

  • Browser-first, privacy-first. Every tool on the site runs client-side in the browser where the architecture allows. File contents never leave the user's device unless explicitly required by the task (e.g. short-URL sharing). This is documented per-tool in each FAQ.
  • Task-complete content, not SEO padding. Every tool page answers the user's original search query in the first fold: what the tool does, the three steps to use it, and one concrete trust note. Long-form expansions sit below the tool UI for readers who want depth, not to please keyword density targets.
  • Append-only content revisions. Published guides and tool copy are revised by appending new sections, not by rewriting existing paragraphs. Readers who returned via a search-ranking link find the same answer they bookmarked, plus anything new.
  • Dated review signals. Every guide and every hub page carries a dateModified / lastReviewed field that is actively maintained. When the content is re-examined (quarterly for evergreen pages, monthly for active clusters) the dates move. Dates that do not move are not trustworthy.

Editorial policy

Our editorial policy is intentionally short and operational: we want it auditable in one screen, not buried across a policy PDF. The full workflow is described in the content-review process below; the two bright-line commitments are:

  • No upload by default. Every tool that can run client-side does run client-side. The exception is short-URL sharing, which is documented per-tool in the FAQ. If a tool needs a backend, the FAQ says so in the first answer.
  • No rewriting of published paragraphs. Corrections are appended, not silently overwritten. Readers who bookmarked a guide via a search result find the same answer they bookmarked, plus any newer addenda.

Editorial cadence

The team operates on a weekly review cadence with monthly milestone cycles:

  • Daily: Monitor Search Console and GA4 dashboards for ranking drops, CTR anomalies, and indexation regressions. Triage against the backlog the same day.
  • Weekly: One cluster gets an editorial sweep (FAQ anti-duplication audit, Tool→Guide link updates, tool page depth review). Cycles across the 8 clusters (ZIP, image-editing, image-conversion, pdf, developer, video, device-test, utility) in rotation.
  • Monthly: Comprehensive phase-level plan authored from the latest SEO analysis. The most recent phase is Phase 11 (2026-04-24); plan files live at seo-reports/<date>/IMPLEMENTATION_PLAN_COMPREHENSIVE.md in the public repository.
  • Quarterly: Full-site rendered audit at all three viewports (390 / 1440 / 1920) on every published route. Freshness bump on every guide's dateModified.

How content is reviewed before it ships

Every tool copy change, guide, and FAQ Q&A passes a three-step review before landing on production:

  1. Anti-duplication audit. Every new paragraph is compared against existing tool-page BODYHTML, BODYWELCOME, FAQ Q&As, and all 22+ published guides for content in the same cluster. Duplicates are either rewritten or removed before commit.
  2. Rendered-browser QA. Every touched route is run through Playwright on mobile-390 and desktop-1440 with ?noads=1 to catch regressions in layout, horizontal scroll, and left/right rail alignment.
  3. Staging soak. Every change ships to staging first, where it is verified live before being mirrored to production. Production commits land on a seo-boost branch; the merge to main is a separate owner action per our hard rule.

Corrections and contact

Site architecture and publication model

The site is a static HTML export generated from a Node-based exporter. Tools are browser-client applications served as static HTML+JS; there is no server-side processing of user file content. The full source and build tooling is public; the editorial workflow (staging-first, append-only, cluster-scoped) is documented in CLAUDE.md and in the homepage footer links.

Questions we have been asked

Who writes the content on each tool page?

Tool copy is authored by the content + SEO editorial function and technically reviewed by the tool engineering function before ship. Guides carry bylines via the editorial-byline.html fragment, which is injected on every guide and hub page. The editorial team is represented collectively as a named Person entity in the structured data (see the Organization JSON-LD on any page).

How do you decide what tool to build next?

Three inputs drive the roadmap: Search Console's impression / click / position signals (what users are searching that we do not yet serve), user correction reports via the contact form, and the running SEO implementation plan (seo-reports/<date>/IMPLEMENTATION_PLAN_COMPREHENSIVE.md). New tools are proposed via the plan, designed, built on staging, vetted through the three-step review above, and then mirrored to production on a separate branch.

Is the content AI-generated?

Content is authored by human editors using AI tooling to accelerate the authoring workflow (drafting, anti-duplication checking, structural outline proposal). Every published paragraph is reviewed by a human editor for factual accuracy, tone, and relevance before commit. We do not publish AI output unreviewed; we treat AI as a co-author and force-multiplier, not a ghostwriter.

Where can I read the editorial plan for the current month?

The live plan and progress files are in the seo-reports/ directory of the site's public source repository, organized by date. The current phase is Phase 11 (authored 2026-04-24) - the plan file covers scope, priority tiers, and the per-item status tracker. The progress file is updated after every cycle ships; it is the source of truth for execution status.
