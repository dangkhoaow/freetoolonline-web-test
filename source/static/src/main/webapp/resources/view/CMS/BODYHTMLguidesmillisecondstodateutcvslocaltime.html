Why the same millisecond timestamp shows different dates

A Unix millisecond timestamp is a pure number: it counts the milliseconds elapsed since the Unix epoch, which is fixed at 1970-01-01T00:00:00 UTC. The timestamp does not carry a timezone, a calendar, or a daylight-saving rule. Every system that displays it picks its own timezone for the human-readable output - your browser uses the operating system's locale, a server uses its container's locale, a SQL database uses its TIMEZONE session setting, and a JSON log uses whatever the writer formatted the value with. The result: the same number can display as 11 different times depending on where the reader is sitting. The four values you should always read together

  • The original number (e.g. 1713538800000) - timezone-free. Store this in databases, send it across HTTP APIs, log it. It never drifts.
  • UTC date/time (e.g. 2024-04-19T15:00:00Z) - the canonical human-readable form. Use this in audit logs, server-side reports, and anything shared across teams in different countries.
  • Local date/time (e.g. 2024-04-19 22:00:00 +07:00 in Bangkok) - what the end user sees on their device. Use this for user-facing emails, calendar invites, and any display that should match the reader's clock.
  • ISO 8601 with offset (e.g. 2024-04-19T22:00:00+07:00) - the round-trip-safe form. Includes both wall-clock time AND offset, so a downstream parser can always recover the exact moment. Prefer this to "April 19, 2024, 10:00 PM" which is ambiguous without a timezone label.

If your tooling only shows one of these four, you cannot tell whether a 7-hour discrepancy is a bug or a timezone difference. The converter on this site shows all four side by side so the comparison is one glance away.

The 30-second sanity check before you ship a date

Whenever a converted date "looks wrong", run this check before debugging the timestamp itself:

  1. Confirm the input is in milliseconds, not seconds. A 13-digit number (like 1713538800000) is milliseconds; a 10-digit number (like 1713538800) is seconds. The two interpretations differ by a factor of 1000 - the same digits read as seconds resolve to a date in 56311 AD. See long number - millisecond or second? for the disambiguation rules.
  2. Note the UTC value the converter returns. This is the canonical, timezone-free answer.
  3. Compute the expected local time from UTC: add the local offset (e.g. +07:00 for Bangkok, -08:00 for Los Angeles, with daylight-saving adjustments). The local value the converter returns should match.
  4. If the local value disagrees, your operating system's timezone is set differently than you assumed - the timestamp is not wrong, the OS is. Adjust the OS or interpret the UTC value directly.

Following these four steps for every conversion catches the two failure modes that produce 99% of "off by hours" bug reports: confusing seconds with milliseconds, and confusing UTC with local time.

When to use UTC vs local in practice

Use UTC for storage and transport. Database columns, JSON over HTTP, log files, audit trails. UTC values do not drift across daylight-saving transitions and do not vary by deployment region. Future readers - including your future self at 3 AM debugging an outage - thank you for storing UTC.

Use local time only for display. Calendar invites, user-facing emails, reminders, "your appointment is at 2 PM". Convert to local at the rendering layer, not in storage. The same database row should render as 22:00 Bangkok time for a Bangkok customer and as 08:00 Los Angeles time for an LA customer - that is correct, not a bug.

Use ISO 8601 with offset for the in-between cases. When you need to send a value to a downstream system that may not know the user's timezone, prefer the offset-included ISO form so the downstream parser can choose how to display it. Avoid free-text "April 19 at 10 PM" - it is unparseable.

Frequently asked questions

Why does the converter show two different times for the same input?
The two are the same instant in different timezones. The number you pasted is timezone-free; the UTC value is the canonical render and the local value is the same instant viewed from your operating system's timezone. The difference (e.g. 7 hours) is the local offset.

The local time looks wrong - how do I fix it?
The local time is whatever your operating system reports as the current timezone. If your laptop is set to Singapore but you expect Bangkok, the local value will be one hour off. Adjust the OS timezone, or interpret the UTC value directly and apply the offset you actually want.

Does daylight saving change the conversion?
Yes - for any timezone that observes DST, the local offset shifts by one hour twice a year. The UTC value never changes. If you compare two timestamps from different sides of a DST boundary, they will look one hour apart in local time even when the elapsed milliseconds are correct.

Can I store local time directly in my database?
Yes, but it is rarely a good idea. Storing UTC and rendering local at the read site is the canonical pattern; it survives server moves, region changes, and DST transitions. Storing local time forces every reader to know which timezone the writer was in - and that information is usually lost.

Is anything sent to a server?
The conversion runs in your browser; the millisecond input and the resulting date/time strings stay on your device. This matters when the timestamp identifies a sensitive event you do not want to log to a third-party server.
