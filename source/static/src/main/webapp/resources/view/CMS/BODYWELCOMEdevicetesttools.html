If a test shows no signal, walk this five-row checklist before assuming the hardware is broken

Most camera, microphone, keyboard, or LCD failures on this hub resolve in one of five places: the browser's site-level permission for that sensor (look for the camera / microphone icon in the address bar), an HTTPS origin (Web APIs refuse to start on plain HTTP), the operating system's privacy panel for the device, another app already holding the sensor open, or - on phones - the iOS / Android sensor-permission prompt that only fires after the first user tap. Work the checklist top-to-bottom; the test re-runs the moment the permission flips.

When a test shows no signal, work the troubleshooting checklist top-to-bottom

When a test shows no signal, work the troubleshooting checklist top-to-bottom - browser permission, HTTPS origin, OS-level privacy panel, device-in-use-by-another-app, mobile sensor quirks - most failures resolve in the first three rows. Camera Test on a black frame, Microphone Test on a flat peak meter, Keyboard Test ignoring a key, or LCD Test refusing to expand to fullscreen all share the same five-row checklist. Browser permission flips first (the camera or microphone icon in the address bar names the site-level permission for that sensor), HTTPS origin verifies the page can call the Web API at all, the OS-level privacy panel grants the browser access to the device, another app holding the sensor open blocks the new request, and on phones the iOS or Android sensor-permission prompt only fires after the first user tap. The test re-runs the moment the permission flips, so most failures resolve without reloading.

Camera test on a laptop: the same checklist works on a built-in webcam

A laptop camera test on the built-in webcam follows the same flow as the standalone Camera Test page above - the browser asks the operating system for the integrated webcam, the OS grants the live frames through the browser permission prompt, and the <video> element draws the preview without any download or sign-up. Built-in laptop webcams almost always register as the default device, so the first Start Camera tap surfaces the lens immediately. If a privacy shutter is closed, the preview stays black even after permission is granted - slide the shutter and the live frames resume in the same tab.

Mobile failures need a different rescue: tap-then-prompt and home-screen-install

On phones the same five-row checklist applies, but two extra rows decide whether the prompt ever appears. iOS Safari blocks the very first sensor read until a real user gesture (a tap on the page) reaches the JavaScript handler - so opening this hub via a deep link or an in-app browser frame can suppress the permission dialog entirely; the fix is one explicit tap on the Start button. Android Chrome occasionally gates hardware-backed flags behind installing the page to the home screen (Add to Home Screen from the menu), after which the same tests re-run with the additional sensor permissions exposed. If the desktop checklist (browser permission, HTTPS origin, OS-level privacy panel, device-in-use-by-another-app, mobile sensor quirks) returns clean and the phone still shows no signal, those two mobile-only rows are almost always the cause.
