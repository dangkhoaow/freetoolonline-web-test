The exact 33 percent expansion is a property of the encoding itself: the algorithm reads three input bytes as one 24-bit unit, splits that unit into four 6-bit chunks, and maps each chunk to one of 64 printable ASCII characters - the uppercase and lowercase Latin letters, the digits 0 through 9, and two padding-safe symbols. The output length is always ceiling(input_bytes / 3) times four bytes, with one or two "=" characters padded onto the tail when the input length is not a clean multiple of three.

The 4 KB rule of thumb that the decision tables above lean on is an HTTP/1.1-era heuristic for inline images in CSS data URLs - on HTTP/1.1 each separate request paid roughly 1 RTT plus header overhead, so under 4 KB an inline image was cheaper than a second request even with the base64 expansion. On HTTP/2 and HTTP/3 the request cost falls (header compression, multiplexing on one connection), and the break-even point shifts up: a separate file is faster sooner. The decision rule still holds in shape - small inline binary in a text destination is the right call; large binary that has a real binary path is the wrong call - only the exact threshold moves with the underlying protocol.

If you have read this far and still want the one-line summary: base64 turns binary bytes into plain ASCII text at a fixed 33 percent size cost, so it is the right call when the destination only accepts text (a CSS data URL, an email body, a JSON field) and the payload is small enough that the bloat does not matter. It is the wrong call when the payload is large, when the destination already supports binary, or when you were reaching for it as a security layer - base64 is reversible by anyone and any decoder unwraps it in one click. For the actual conversion, open Image to Base64 or Base64 to Image.

The two in-browser conversion tools split the work along the direction you need: encoding bytes into a data URL, or decoding a string back into a downloadable image. For the actual conversion in the encoding direction, open Image to Base64 - you pick a file on disk and the tool returns the encoded ASCII string, ready to paste into a CSS data URL, an email body, or a JSON field. For the decoding direction, open Base64 to Image - you paste the encoded string and the tool decodes it back into a downloadable image file. Both tools run the conversion locally in the browser without uploading the file to a server, which matters when the image is private or pre-release.

The four most common right calls share the same constraint: the destination only accepts text, and the payload is small. A tiny icon embedded in a CSS data URL saves one HTTP request and keeps the icon styled alongside its selector. A binary email attachment routed through MIME is wrapped in base64 so it survives a 7-bit transport. Binary bytes carried inside a JSON field travel as plain text because the protocol cannot store raw bytes natively. And a short binary identifier dropped into a URL uses the URL-safe variant so the encoded string contains no slash or plus character that the path or query layer would mangle.

The four common wrong calls are equally easy to spot once you have the rule. Stuffing a 500 KB photo into a CSS data URL means the file cannot be cached separately and blocks the parser until the whole stylesheet finishes downloading. Treating base64 as a security layer fails on first contact: any decoder reverses it in one click. Storing binary in a database TEXT column when a BLOB type exists loses the binary-column affordances the schema already provided. And shipping multi-megabyte data over base64 when a streaming binary upload would be cheaper just doubles the bandwidth bill for no encoding payoff.
