Apa arti preflight sebenarnya

"Preflight" dipinjam dari pemeriksaan pra-penerbangan penerbangan dan masuk ke dunia PDF melalui Adobe pada awal 2000-an, pertama di InDesign lalu di Acrobat. Ini adalah langkah validasi pra-cetak / pra-publikasi yang menghasilkan LAPORAN yang mendaftar masalah struktural yang dimiliki file PDF terhadap standar atau alur kerja target. Masalah tipikal yang dilaporkan pemeriksaan preflight: font hilang, font tidak tertanam, gambar RGB dalam alur kerja CMYK-only, gambar resolusi rendah di bawah DPI target, transparansi tidak diratakan, enkripsi (yang memblokir penerimaan di beberapa sistem arsip), tindakan JavaScript (yang merusak arsip PDF/A), dan cacat struktural PDF.

Preflight adalah langkah PEMERIKSAAN. Tidak MENGATASI file. Pembaca (atau alat hilir) membaca laporan dan memperbaiki file di alat layout / penulisan hulu, atau menjalankan langkah perbaikan terpisah.

Apa yang sebenarnya diperiksa /preflight-pdf.html

Alat preflight freetoolonline melaporkan dua level konformitas arsip spesifik: PDF/A-1A (varian teks-dapat-diakses / dengan tag) dan PDF/A-1B (varian hanya kesetiaan visual). PDF/A adalah format arsip ISO 19005 yang digunakan oleh arsip nasional, perpustakaan, dan alur kerja apa pun yang membutuhkan PDF dirender dengan cara yang sama beberapa dekade dari sekarang. Aturan strukturalnya didefinisikan dengan baik:

  • Font harus tertanam sebagai subset Type 1 / TrueType / OpenType / CFF / Type 3, sehingga file tidak bergantung pada ketersediaan font pada renderer masa depan.
  • Tanpa transparansi pada halaman (harus diratakan menjadi pixel komposit dulu).
  • Tanpa enkripsi (file harus dapat dibuka tanpa kata sandi).
  • Tanpa video tertanam / tanpa JavaScript tertanam / tanpa referensi file eksternal (file harus mandiri untuk arsip).
  • Ruang warna harus independen perangkat (ICC-tagged, bukan RGB atau CMYK perangkat mentah tanpa output intent).

Unggah PDF ke Preflight PDF, dan laporan memberi tahu Anda aturan struktural mana yang dilewati atau gagal file Anda terhadap PDF/A-1A dan PDF/A-1B.

Apa yang TIDAK diperiksa

Alat ini jujur tentang cakupannya. Daftar di bawah adalah apa yang TIDAK dilakukan /preflight-pdf.html, dan kemana harus pergi.

  • Tidak MEMPERBAIKI masalah. Preflight adalah langkah pemeriksaan. Melaporkan masalah; tidak menyematkan font yang hilang, tidak meratakan transparansi, tidak mengonversi RGB ke CMYK. Untuk memperbaiki masalah, langkah berikutnya yang tepat adalah mengekspor ulang dari alat layout / penulisan sumber dengan pengaturan yang dikoreksi, atau gunakan alat perbaikan offline.
  • Tidak melakukan preflight cetak PDF/X. PDF/X-1a (CMYK-only, tanpa transparansi, tanpa masalah warna spot), PDF/X-3 (mengizinkan RGB dengan tag ICC), dan PDF/X-4 (mengizinkan transparansi aktif) adalah standar ISO 15930 yang berbeda dari PDF/A dan memerlukan analisis sadar-CMYK. Untuk konformitas PDF/X, alat yang tepat adalah mesin preflight lengkap Adobe Acrobat Pro (offline), atau alur kerja preflight percetakan.
  • Tidak memvalidasi keabsahan hukum tanda tangan digital. Tanda tangan PDF bisa kriptografis valid (byte cocok dengan hash yang ditandatangani) tetapi secara hukum tidak valid (sertifikat penandatangan tidak ada di trust store CA Anda, atau waktu penandatanganan di luar jendela validitas sertifikat). Untuk verifikasi keabsahan hukum tanda tangan, alat yang tepat adalah Adobe Acrobat atau validator tanda tangan resmi yurisdiksi Anda.
  • Tidak memperbaiki PDF rusak. Jika PDF Anda tidak terbuka (pembaca PDF mengatakan "file rusak" atau "tidak dapat membuka file ini"), aliran byte rusak dan preflight kemungkinan akan menolak membaca file sama sekali. Langkah berikutnya yang tepat adalah alat perbaikan PDF yang merekonstruksi tabel referensi silang.
  • Tidak memvalidasi aksesibilitas PDF/UA. PDF/UA (Universal Accessibility, ISO 14289) adalah standar PDF tagged untuk kompatibilitas dengan pembaca layar dan teknologi bantu. Lebih ketat daripada persyaratan tagging PDF/A-1A. Untuk validasi PDF/UA, validator aksesibilitas khusus adalah alat yang tepat.

Tabel terjemahan kosakata pencarian

Pencarian yang berbeda menggunakan kata yang berbeda untuk konsep yang sama (atau untuk konsep yang berbeda). Tabel di bawah memetakan kosakata pencarian umum ke apakah /preflight-pdf.html adalah alat yang tepat.

  • "validator pdf online" / "verifikasi pdf" / "periksa pdf" → YA, jika pertanyaannya adalah konformitas arsip PDF/A. Preflight PDF menjawab ini.
  • "preflight pdf online" / "preflight pdf" → YA untuk PDF/A. TIDAK untuk preflight cetak PDF/X.
  • "ratakan pdf online" / "flatten pdf" → TIDAK. Pemerataan adalah operasi berbeda: mengunci bidang formulir dan anotasi di halaman sehingga tidak dapat lagi diedit. Gunakan Flatten PDF untuk itu.
  • "perbaiki pdf" / "fix pdf" → TIDAK. Preflight melaporkan masalah; tidak memperbaikinya. Untuk perbaikan PDF rusak, gunakan alat rekonstruksi khusus.
  • "pdf/x preflight" / "preflight cetak" → SEBAGIAN. Alat freetoolonline memeriksa PDF/A, bukan PDF/X. Mesin preflight offline Adobe Acrobat Pro adalah alat yang tepat untuk PDF/X-1a / 3 / 4.
  • "verifikasi tanda tangan pdf" → TIDAK. Gunakan Adobe Acrobat atau validator tanda tangan resmi yurisdiksi Anda.

Alur keputusan: dari PDF yang butuh pemeriksaan ke alat yang tepat

Jika Anda memiliki PDF di depan Anda dan tidak yakin apakah /preflight-pdf.html adalah langkah berikutnya yang tepat, alur empat-langkah di bawah merutekan Anda ke alat yang tepat.

  1. Sebutkan pertanyaan sebenarnya. Apakah Anda memvalidasi konformitas arsip (PDF/A)? Konformitas cetak (PDF/X)? Memperbaiki file rusak? Mengunci bidang formulir? Memverifikasi tanda tangan digital? Setiap pertanyaan merutekan ke alat yang berbeda.
  2. Cocokkan pertanyaan dengan alat. Arsip PDF/A → Preflight PDF. Cetak PDF/X → Adobe Acrobat Pro offline. Perbaikan PDF rusak → alat rekonstruksi khusus. Mengunci bidang formulir dan anotasi di halaman → Flatten PDF. Keabsahan hukum tanda tangan → Adobe Acrobat atau validator yurisdiksi.
  3. Lakukan tindakan di alat yang tepat. Unggah ke /preflight-pdf.html untuk arsip PDF/A; atau jelajahi hub yang lebih luas PDF Tools untuk kecocokan terbaik berikutnya.
  4. Baca laporan dan putuskan perbaikan. Preflight menghasilkan laporan. Putuskan apakah akan mengekspor ulang dari alat penulisan hulu (perbaikan paling umum), menjalankan langkah perbaikan terpisah, atau menerima file sebagai tidak konforman untuk standar target.
